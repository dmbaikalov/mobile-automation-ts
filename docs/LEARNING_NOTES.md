# Конспект курсу з мобільної автоматизації

Особистий конспект, який веду під час менторства з мобільної автоматизації (Appium 3 + WebdriverIO + TypeScript). Оновлюється після кожного уроку.

> Курс орієнтується на актуальні мажорні версії інструментів на момент навчання, а не на версії, зафіксовані в початковому технічному завданні — стек перевіряється й оновлюється, коли з'являється новіша стабільна версія.

---

## Фаза 1 — Основи мобільної автоматизації

### Урок 1: Навіщо існує Appium

**Типи мобільних застосунків:**
- **Native** — Kotlin/Java (Android) або Swift/Obj-C (iOS), рендериться самою ОС, DOM відсутній.
- **Hybrid** — нативна оболонка + WebView усередині (наприклад, checkout-екран).
- **Mobile web** — просто Chrome/Safari на телефоні.

**Чому Playwright-підхід не працює на мобільному:**
Playwright напряму контролює рушії браузера (Chromium/WebKit/Firefox), бо це open-source проєкти з протоколами для зовнішнього керування (CDP). UiAutomator2 (Android) і XCUITest (iOS) — закриті automation-фреймворки, вбудовані в ОС Google/Apple. Немає доступу "всередину" — тільки через офіційні automation API, які надає сама ОС.

**Роль Appium:** сервер, що приймає WebDriver-команди від тесту і **транслює** їх у виклики, специфічні для платформного драйвера (не просте проксіювання — саме трансляція протоколу в інший API).

```
WebdriverIO тест → Appium Server → UiAutomator2 / XCUITest драйвер → Android/iOS automation API → застосунок
```

**Чому мобільна флакі-поведінка гірша за веб:**
На вебі є детерміновані сигнали готовності (DOM ready, network idle, mutation observer). На мобільному таких стандартизованих сигналів немає — UI-дерево будує нативний рендерер ОС, і Appium дізнається про появу елемента лише постфактум, коли він фізично з'явився в дереві. Додатково: анімації переходів, CPU throttling, фонові процеси ОС на реальному пристрої/емуляторі вносять більше змінних, ніж контрольоване середовище headless-браузера. → Тема глибше розкриється у Фазі 8 (синхронізація).

**Навіщо перемикатись у web context на hybrid-екранах:**
Нативні локатори (UiAutomator2) бачать WebView лише як один непрозорий елемент — структури всередині не видно взагалі. Перемикання в web context — це не зручність, а технічна необхідність: без нього немає фізичного доступу до елементів усередині WebView. Після перемикання працюють звичні CSS/XPath-селектори.

**Ключовий висновок:** Appium — не сам automation-рушій, а шар трансляції. Флакі-поведінка й баги найчастіше йдуть від нижчого драйвера (UiAutomator2/XCUITest) або самої ОС, а не від Appium напряму.

## Фаза 2 — Android екосистема

### Урок 2: Складові Android-екосистеми та вибір емулятора

**Ключові частини:**
- **Android SDK** — інструменти для розробки/запуску Android (emulator, adb, платформні образи).
- **Android Studio** — тут використовується як GUI-обгортка над SDK Manager і Device (AVD) Manager.
- **AVD (Android Virtual Device)** — віртуальний пристрій: обираєш device profile, system image (API level), ABI. Аналог `playwright install chromium`, але замість бінарника браузера — цілий віртуальний Android-пристрій конкретної версії ОС.
- **adb (Android Debug Bridge)** — CLI для спілкування з пристроєм/емулятором (встановлення APK, `logcat`, shell). Appium використовує adb під капотом.
- **APK** — файл застосунку для встановлення й тестування (аналог відкриття URL у вебі).
- **Emulator (Android) vs Simulator (iOS)** — НЕ взаємозамінні терміни. Android emulator — справжня віртуалізація (QEMU). iOS simulator — симуляція на рівні API macOS, без повної віртуалізації заліза. Практичний наслідок: нативні бінарні залежності можуть поводитись по-різному.

**Чому для тестового стенду обирають стабільний API level (напр. 33/34), а не найновіший:**
1. **Відповідність реальному розподілу користувачів** — головна причина. Мета тестування — поведінка, з якою реально стикаються користувачі. Тестувати на API, якого ще нема на реальних пристроях, означає перевіряти неіснуючі сценарії і пропускати реальні.
2. **Стабільність драйверів автоматизації** — UiAutomator2/Appium патчаться переважно проти вже випущених API. Preview/найновіші API можуть мати непропатчені баги в accessibility-шарі ОС, які виглядають як флакі тест, а насправді баг платформи.
3. **Відтворюваність у CI** — CI-раннери (напр. `reactivecircus/android-emulator-runner`) мають перевірені, кешовані images для усталених API levels; найновіші/preview можуть бути недоступні або нестабільні в CI.

**Практичний висновок для курсу:** основний AVD для решти курсу — Pixel, API 33 (Android 13), Google APIs image (без Google Play Store, бо Play Services не потрібні для чистої Appium-автоматизації).

**Типові помилки:**
- Плутати emulator (Android) і simulator (iOS) як синоніми.
- Обирати system image навмання, не звертаючи уваги на Google APIs vs Google Play варіант.
- Після зміни конфігурації AVD не перезапускати його повністю (cold restart) — Android Studio попереджає про це банером "Changes will apply on restart".

### Урок 3: adb — базова діагностика

**`adb devices`** — список підключених пристроїв/емуляторів:
```
List of devices attached
emulator-5554   device
```
- `emulator-5554` — serial пристрою (порти емуляторів завжди парні, починаючи з 5554); знадобиться як `udid` capability в Appium-конфігу, коли пристроїв кілька одночасно.
- Статус `device` = готовий. Інші статуси: `offline` (завис), `unauthorized` (не підтверджено USB debugging на реальному пристрої).
- Якщо Appium-конфіг не вкаже явний `udid` при кількох підключених пристроях, буде обрано перший зі списку — часта причина "тест запустився не на тому пристрої" в командних середовищах.

**Формат рядка `logcat`:**
```
07-11 11:17:18.656  1758  1758 I AiAiTextClassifier: повідомлення
   дата+час          PID   TID  рівень   тег              текст
```
Рівні логування: `V < D < I < W < E < F` (Verbose → Fatal), той самий принцип severity, що і в звичайних логерах.

**Ключовий висновок:** повний `logcat` (і навіть відфільтрований по `*:E`) на чистому емуляторі завжди показує велику кількість помилок від системних Google-сервісів (Google Assistant, GMS `constellation`, mock HAL-сервіси на кшталт `power.stats`, `SettingsToPropertiesMapper`) — вони намагаються зв'язатись з хмарою/залізом, якого немає на емуляторі. **Це нормально й не стосується жодного застосунку.**

**Професійне правило:** ніколи не читати весь logcat "як є". Завжди фільтрувати за PID або тегом конкретного тестованого застосунку:
```
adb -s emulator-5554 logcat -d --pid=$(adb -s emulator-5554 shell pidof -s com.example.app)
```
Усе решта — фоновий системний шум, який існує завжди, незалежно від того, що тестується.

## Фаза 3 — Архітектура Appium

### Урок 4: Client-Server архітектура

**Ключова відмінність від Playwright:** у Playwright немає окремого сервера — `chromium.launch()` керує браузером у тому ж/дочірньому процесі прозоро для користувача. В Appium **сервер — окремий, самостійний Node.js процес**, який слухає HTTP-порт (типово `4723`) і має бути запущений *до* старту тестів.

```
Тест (WebdriverIO client) ──HTTP (WebDriver protocol)──▶ Appium Server ──▶ Driver (UiAutomator2/XCUITest) ──▶ Emulator/Device
```

**Появу цієї архітектури "сервер + окремі драйвери-плагіни" ввели ще в Appium 2.x**, і вона збереглась і розвинулась у поточній **Appium 3.x**, яку ми використовуємо в курсі: сервер — це ядро-диспетчер; драйвери (UiAutomator2, XCUITest) — окремі npm-пакети, встановлюються окремо (`appium driver install uiautomator2`), не вбудовані в сам Appium. Аналог — `playwright install <browser>`, тільки тут йдеться про цілі драйвери автоматизації.

**Сесія (Session):** тест відправляє `POST /session` з `capabilities` (платформа, пристрій, застосунок) → сервер повертає унікальний `sessionId`. Кожна наступна команда (`findElement`, `click`) іде з цим `sessionId` у шляху: `POST /session/{sessionId}/element`. Концептуально схоже на `browser.newContext()` у Playwright, але "контекст" тут — фізичний/віртуальний пристрій.

**Один сервер може обслуговувати кілька паралельних сесій одночасно** — сервер тримає внутрішню мапу `sessionId → інстанс драйвера → конкретний пристрій (udid)` і маршрутизує команди за `sessionId` (аналог session token на бекенді, що асоціює запит із користувачем, а не з конкретним з'єднанням).

**Два підходи до паралелізації:**
| Підхід | Коли використовують |
|---|---|
| 1 сервер, N сесій (мультиплексування через `sessionId`) | Стандарт у CI/хмарних сервісах (BrowserStack, Sauce Labs) — менше накладних витрат |
| N окремих серверів | Потрібна повна ізоляція процесів (різні версії драйверів одночасно) або сервер сам став вузьким місцем через event-loop Node.js при дуже високому навантаженні |

**Типові помилки:**
- Забути запустити Appium Server перед тестами → `ECONNREFUSED`, що новачки плутають з помилкою в тестовому коді.
- Плутати "Appium" як моноліт з "Appium + конкретний драйвер" — проблеми з iOS-елементами зазвичай стосуються XCUITest-драйвера, а не Appium Server як такого.

## Фаза 4 — Налаштування середовища

### Урок 5: Встановлення Appium Server і драйвера

Встановлено: Appium **v3.5.2** — актуальна мажорна версія на момент курсу (новіша за "Appium 2.x", яка згадувалась як орієнтир на старті; архітектура клієнт-сервер та сама, але деталі API варто перевіряти саме проти v3 доків), драйвер `uiautomator2`.

**`appium driver doctor uiautomator2`** — обов'язкові перевірки (`ANDROID_HOME`, adb/emulator, `JAVA_HOME`) усі пройшли. Опціональні попередження (`bundletool.jar`, `ffmpeg.exe`, GStreamer) — не блокери, не потрібні прямо зараз:
- `bundletool.jar` — для Android App Bundles (`.aab`), ми працюємо з `.apk`.
- `ffmpeg.exe` — запис відео тесту, знадобиться пізніше (діагностика падінь).
- GStreamer — live-стрімінг екрана пристрою, малоймовірно знадобиться в курсі.

**Правило:** дивитись на підсумковий рядок doctor-перевірки (`"0 required fixes needed"`), а не панікувати від кожного `✖` — оптональні попередження не блокують роботу.

**Java-версія:** `JAVA_HOME` вказує на JDK 25 — значно новіша за типові LTS (11/17), з якими найкраще протестовані Android build tools/Gradle. Поки не проблема, але перше, що варто перевірити при дивних помилках збірки в майбутньому.

**Запуск сервера (`appium`):** слухає `http://0.0.0.0:4723`. У виводі автоматично завантажились **два** драйвери — `uiautomator2` (встановлений явно) і **`chromium`** (підтягнувся автоматично).

**Навіщо потрібен chromium-драйвер:** Android WebView побудований на Chromium-рушії. Коли Appium перемикає контекст з `NATIVE_APP` на `WEBVIEW_*` (Урок 1 — hybrid-екрани типу checkout), UiAutomator2 фізично не бачить структуру всередині WebView — тому керування всередині WebView-контексту делегується chromium-драйверу, який використовує CDP (Chrome DevTools Protocol) — той самий протокол, яким керує Playwright. Це прямий технічний наслідок архітектури з Уроку 1, не окрема "фіча про всяк випадок".

**Спостереження:** глобальне `~/.appium` середовище вже мало встановлені плагіни (`device-farm`, `images`) з попередніх експериментів — не активовані (`No plugins activated`), не заважають роботі.

## Фаза 4/5 — Перший проєкт і перший тест

### Урок 6: Ініціалізація WebdriverIO-проєкту, capabilities, перший запуск

**Ініціалізація:** `npm init wdio@latest .` — конфігураційний майстер (аналог `npm init playwright@latest`). Ключові вибори для мобільного курсу: E2E Testing, "On my computer", Mobile → Android, Mocha, TypeScript, reporters `spec` + `allure`, service `appium`, без сторонніх плагінів (`wait-for`, testing-library — не потрібні, WDIO вже дає `waitUntil()` з коробки; власні утиліти напишемо самі у Фазі 8), без Visual Testing (окрема тема, Фаза 11), без `appium-installer` (уже все налаштовано вручну й перевірено `doctor`-командою).

**Демо-код майстра орієнтований на веб, не мобільний застосунок:** згенеровані `test/pageobjects/login.page.ts`, `secure.page.ts`, `page.ts`, `test/specs/test.e2e.ts` — приклад для `the-internet.herokuapp.com` (публічний навчальний сайт для Selenium/WebdriverIO). `Page.open()` викликає `browser.url()` — команда навігації по URL, специфічна для web-контексту; у native мобільному тесті немає URL, натомість застосунок просто запускається через `appium:app` capability. **Ці файли видалені** — не мають цінності як робочий код для нашого курсу.

**Виправлення capabilities під native Android:**
```ts
capabilities: [{
  platformName: 'Android',
  'appium:app': process.env.ANDROID_APP_PATH,
  'appium:deviceName': 'Pixel 7 Pro (2)',
  'appium:platformVersion': '13.0',
  'appium:automationName': 'UiAutomator2',
}]
```
- `browserName: 'Chrome'` (дефолт майстра) прибрано — це capability для мобільного web-тесту, не для native APK.
- `appium:app` має вказувати на **конкретний файл** (`.apk`), не на теку — інакше Appium не може резолвити встановлюваний артефакт.
- Значення `appium:app` читається з `process.env.ANDROID_APP_PATH`, а не хардкодиться рядком у конфігу — бо це значення відрізняється між середовищами (локально/CI/оновлена версія APK), і межа між конфігурацією середовища й логікою коду — це і є сенс існування `.env`-паттерну.

**`.env` автопідхоплюється сучасним WebdriverIO CLI (v9) без явного `dotenv.config()`** — перевірено практично: `process.env.ANDROID_APP_PATH` зрезолвився і сесія піднялась, хоча `dotenv` не викликався явно в коді. Причина — `dotenv` присутній у `node_modules` як **транзитивна залежність** (тягнеться через `@wdio/cli`/суміжні пакети), і CLI сам читає `.env` з кореня проєкту.

**Урок про "phantom dependencies":** те, що щось "просто запрацювало" завдяки транзитивній залежності — не привід залишати це неявним. Якщо поведінка коду залежить від пакета, цей пакет має бути **явною** залежністю в `package.json` (`npm install dotenv --save-dev`), інакше майбутнє оновлення `@wdio/cli` може мовчки прибрати цю транзитивну залежність і непередбачувано зламати `.env`-механізм без жодного натяку в `package.json` чому.

**Перший реальний запуск (`npm run wdio`) з sanity-тестом (`driver.getPageSource()`):**
- Спочатку отримали `No specs found to run` — не помилка конфігурації, а прямий наслідок видалення демо-спеку без створення нового; підтвердило, що сама інфраструктура (Appium Server підйом через `@wdio/appium-service`, зчитування конфігу) працює коректно.
- Після додавання мінімального spec-файлу — сесія успішно піднялась на реальному емуляторі, застосунок `com.wdiodemoapp` встановився й запустився, `getPageSource()` повернув реальне XML native UI-дерево з кнопками `Home/Web/Login/Forms/Swipe/Drag/Menu` — це офіційний WebdriverIO demo-застосунок, зручний для наступних уроків (форми, свайпи, логін).

**`npm audit` — дисципліна, а не автоматичне виправлення:** після встановлення `dotenv` виявлено вразливість у `serialize-javascript` (транзитивно через стару `mocha` в ланцюжку `@wdio/mocha-framework`). `npm audit fix --force` запропонував **понизити** `@wdio/mocha-framework` до v7.x — це суперечило б свідомому рішенню тримати актуальний стек (v9.x). **Правило: ніколи не запускати `--force`-варіант naosліп** — спочатку читати, що саме буде змінено (breaking change?) і чи вразливість реально експлуатовна в контексті локальних/CI тестів (тут — так собі ризик, не обробка недовіреного зовнішнього вводу). Залишено як відоме, прийнятне попередження до апстрім-виправлення.

## Фаза 6 — Локатори

### Урок 7: Стратегії локаторів + Appium Inspector

**Немає CSS/DOM у native мобільній автоматизації** — замість цього атрибути нативного UI-дерева: `content-desc` (accessibility), `resource-id`, `class` (нативний тип компонента), `text`, `bounds`.

**Стратегії локаторів (Android/UiAutomator2), за пріоритетом стабільності:**
1. **Accessibility ID** (`~значення`, тобто `content-desc`) — найстабільніший: навмисно доданий розробниками для UI-тестів і screen readers, семантичний "контракт".
2. **resource-id** — стабільний, зашитий у XML-layout застосунку.
3. **UiAutomator2 selector** (`android=new UiSelector()...`) — Android-специфічний DSL.
4. **XPath** — уникати, коли є альтернатива: на мобільному змушує драйвер обходити все (часто глибоко вкладене, 15+ рівнів) дерево елементів → повільніше; XPath за структурою дерева ламається при найменшій зміні верстки, навіть якщо елемент логічно не змінився.

**Appium Inspector** — десктопний GUI-інструмент (аналог Playwright Inspector/Codegen): підключається до **окремо запущеного** Appium Server (`appium` в терміналі, НЕ через `@wdio/appium-service`, бо той сам гасить сервер після тесту), показує живий скріншот + дерево елементів + атрибути обраного елемента + готові приклади локаторів. Capabilities вводяться в Inspector як **чистий JSON** (без коментарів, `appium:app` — реальний абсолютний шлях на диску, не `process.env...`, бо Inspector не виконує JS/не читає `.env`).

**Пастка: клікабельний контейнер vs вкладений текст/іконка.** Кнопка в native Android UI часто складається з батьківського `Button`/`ViewGroup`-контейнера (реально обробляє клік, `clickable: true`) і дочірніх `TextView`/`ImageView` всередині (візуальні, `clickable: false`). Обираючи елемент в Inspector, важливо клікати саме на клікабельний контейнер — інакше отримаєш крихкий xpath-локатор за текстом замість стабільного accessibility id.

**Пастка: однакові написи, різні ідентифікатори.** У demo-застосунку кнопка `Login` на головному екрані (навігація, `content-desc="Login"`, `class="Button"`) і кнопка `LOGIN` на формі логіну (сабміт, `content-desc="button-LOGIN"`, `class="ViewGroup"`) виглядають схоже візуально, але мають різні accessibility id і навіть різний нативний клас. Підтверджує, чому орієнтуватись на видимий текст ризиковано — потрібні недвозначні ідентифікатори.

**Знайдені локатори demo-застосунку:**
```ts
$('~Login')           // кнопка навігації на головному екрані
$('~input-email')     // поле email на формі логіну
$('~input-password')  // поле password на формі логіну
$('~button-LOGIN')    // кнопка сабміту форми логіну
```

### Урок 8 (передчасно, з практики): `$()` лінивий, findElement асинхронний і паралельний

При запуску тесту з кількома `const el = $('~...')`, оголошеними підряд без `await`, у логах Appium видно **всі** `findElement`-запити, що стартують практично одночасно — включно з пошуком елементів, які на той момент ще не існують на екрані (бо попередній `.click()` для навігації ще не виконався). Це не помилка коду: `$(...)` повертає "лінивий" проміс-подібний об'єкт, фактичний пошук виконується (і за потреби повторюється) у момент реальної дії/перевірки (`.click()`, `.setValue()`, `expect().toBeDisplayed()`).

**Чому тест все одно проходить, попри видимі `no such element` в логах:** `expect(el).toBeDisplayed()` (через `expect-webdriverio`) має вбудований **retry з timeout** (`waitforTimeout` у `wdio.conf.ts`, дефолт 10s) — він повторює пошук, поки елемент не з'явиться або не вийде час. Без цього retry-механізму практично кожен мобільний тест був би хронічно флакі: клік на кнопку навігації запускає перехід/анімацію екрана, і наступна миттєва (без retry) перевірка може виконатись до того, як новий екран встиг відрендеритись — пряме продовження теми з Уроку 1 про відсутність детермінованих сигналів готовності UI на мобільному. Тема розкриється повністю у Фазі 8 (Синхронізація).

## Фаза 7 — Жести

### Урок 9: Незалежність тестів (`beforeEach`/`afterEach`) + свайп відносно елемента

**Незалежність тестів — критичне правило.** Якщо тест A залишає застосунок в іншому стані/екрані, ніж на якому очікує стартувати тест B, B стає залежним від порядку виконання й ламким. Рішення — скидати стан застосунку між тестами:
```ts
beforeEach(async () => { await driver.activateApp(APP_PACKAGE) })
afterEach(async () => { await driver.terminateApp(APP_PACKAGE) })
```
`terminateApp` в `afterEach` (а не тільки `beforeEach`) гарантує чистий стан і після останнього тесту в файлі, не лише між тестами.

**Package name (`com.wdiodemoapp`) — не кандидат у `.env`.** На відміну від `ANDROID_APP_PATH` (змінюється між середовищами), package name — фіксована властивість самого зібраного APK, однакова завжди. Правильне місце — константа в коді (`test/constants.ts`), не змінна середовища.

**Жести через `mobile: swipeGesture` (W3C Actions під капотом, делеговані нативному UiAutomator2 API):**
```ts
const card = await $('~card')
const { x, y } = await card.getLocation()
const { width, height } = await card.getSize()

await driver.execute('mobile: swipeGesture', {
  left: x, top: y, width, height,
  direction: 'left',
  percent: 0.75,
})
```
**Ключове правило:** координати області свайпу мають бути **відносними до реального елемента** (`getLocation()`/`getSize()`), а не захардкодженими абсолютними пікселями. Захардкоджені координати ламаються на інших розмірах екрана/пристрою — той самий принцип "конфігурація відносна, не абсолютна", який вже зустрічався з API level (Урок 2) і шляхами до файлів (Урок 6). Перевірено на практиці: свайп з довільно підібраними координатами (`left: 100, top: 800, width: 800, height: 400`) не спрацював (жест "виконався" без помилки, але нічого не змінилось), бо область не збігалась з реальним розташуванням картки.

**`getRect()` не існує в WebdriverIO API елемента** — типізація (`ChainablePromiseElement`) ловить це на етапі редагування коду ще до запуску тесту. Правильні окремі методи: `getLocation()` → `{x, y}`, `getSize()` → `{width, height}`.

**UiAutomator2 selector для пошуку за частковим текстом**, коли елемент не має власного accessibility id (лише вкладений `text`):
```ts
$('android=new UiSelector().textContains("GREAT COMMUNITY")')
```
Використано для перевірки, що після свайпу з'явилась **наступна** картка каруселі (текст, реально побачений в Appium Inspector після виконання свайпу, а не вгаданий заздалегідь).

**Appium Inspector, вкладка Gestures** — дозволяє виконати свайп/жест прямо в інструменті для діагностики, перш ніж писати код, замість підбору координат наосліп.

## Фаза 8 — Синхронізація

### Урок 10: `expect().toX()` vs `waitUntil()` — різні контракти

**Три рівні синхронізації в WebdriverIO:**
1. `expect(el).toX()` (`expect-webdriverio`) — вбудований retry до `waitforTimeout`, для перевірки стану елемента.
2. `browser.waitUntil(callback, opts)` — низькорівневе очікування **довільної булевої умови**, не обов'язково пов'язаної з елементом.
3. Автоматичний retry перед діями (клік/пошук) — схоже на Playwright auto-waiting, але "елемент існує" ≠ "дія дала очікуваний ефект" (див. нижче).

**Найважливіша пастка (побачена на практиці в Уроці 9):** дія може виконатись без винятку, але не досягти мети — свайп у неправильній області "спрацював" (жодної помилки), але нічого не змінилось на екрані. **Правило:** кожна дія зі видимим ефектом (клік, свайп, введення тексту) має супроводжуватись явною перевіркою результату, а не лише відсутністю помилки виконання команди.

**Критична помилка при змішуванні `waitUntil()` з `expect()`:**
```ts
// НЕПРАВИЛЬНО — expect кидає виключення, а не повертає false, waitUntil це не ловить
await browser.waitUntil(async () => await expect(el).toHaveText(text), {...})
```
`waitUntil()` очікує колбек, що повертає **булеве значення**; `expect().toX()` при невідповідності **кидає виключення**, а не повертає `false` — тому `waitUntil` падає з першої ж невдалої спроби, замість повторення. Побачено на практиці: тест падав миттєво, а не після вичерпання timeout.

**Правильний варіант — булева умова без `expect` усередині:**
```ts
await browser.waitUntil(
  async () => (await inputField.getText()) === textToEnter,
  { timeout: 10000, timeoutMsg: '...', interval: 1000 }
)
```
Якщо перевірку можна виразити готовим `expect().toX()`-матчером — використовувати саме його напряму (retry вже вбудований); `waitUntil()` лишати для умов, які не покриваються готовими матчерами.

**`pause()`/`sleep()` з фіксованою затримкою — завжди уникати.** "Магічне число" або надто коротке (тест все одно впаде на повільнішому CI/пристрої), або надто довге (марно гальмує кожен прогін). `waitUntil()`/`expect().toX()` чекають рівно стільки, скільки реально потрібно.

**Дрібниця про API полів вводу:** `getText()` на `EditText` іноді повертає порожній рядок замість введеного значення (залежно від застосунку); спеціалізована альтернатива для input-полів — `getValue()`.

## Фаза 9 — Page Object Model

### Урок 11: Мобільна POM-архітектура, глобальні хуки, конфлікт паралельних сесій

**Чому POM для мобільного відрізняється від Playwright:** "сторінка" — це не URL, а **екран застосунку**, куди потрапляєш через дії (клік по навігації), а не прямий перехід. Base-клас (`BasePage`) не має `open()` через `browser.url()` — немає сенсу для native-застосунку.

**Структура:**
```
test/pageobjects/BasePage.ts     — спільний базовий клас
test/pageobjects/MainScreen.ts   — головний екран, геттери-локатори + методи навігації (openLogin/openForms/openSwipe)
test/pageobjects/LoginScreen.ts  — екран форми логіну, waitForDisplayed() + login(email, password)
```
Кожен Page Object експортується як **готовий singleton-екземпляр** (`export default new MainScreen()`), не клас — сторінка застосунку концептуально одна.

**Розділення відповідальності в Page Object:** `waitForDisplayed()` (перевірка готовності екрана) і `login()` (сама дія) — окремі методи, жоден не викликає інший автоматично. Свідомий архітектурний вибір, не єдино правильний — деякі команди роблять навпаки (кожен публічний метод сам гарантує готовність сторінки).

**У WebdriverIO немає фікстур, як у Playwright** (`test.extend()`, dependency injection). Найближчий аналог для спільної логіки, що застосовується до кожного тесту, — глобальні хуки життєвого циклу в `wdio.conf.ts`: **`beforeTest`/`afterTest`** (framework-агностичні, на відміну від Mocha-специфічних `beforeEach`/`afterEach`, які й далі можна використовувати локально в spec-файлі за потреби чогось нестандартного). Перенесено `activateApp`/`terminateApp` з локальних хуків spec-файлу в глобальні — виправдано, бо це справді універсальна потреба кожного тесту, а не випадковий збіг для трьох наявних тестів.

**Знайдена на практиці проблема паралелізації:** коли в `test/specs/` одночасно існували старий (`login-navigation.e2e.ts`) і новий (`login.e2e.ts`) spec-файли з тим самим функціоналом, WebdriverIO (`maxInstances: 10`) запустив їх **паралельно на різних воркерах**, і обидва намагались керувати **одним і тим самим емулятором** одночасно → `instrumentation process is not running (probably crashed)` і `session ... is not known`. **Висновок:** кілька spec-файлів не можуть паралельно ділити один пристрій — паралелізація в мобільній автоматизації вимагає окремих пристроїв/емуляторів на кожен паралельний потік, не лише окремих контекстів, як у Playwright. Тема повністю розкриється у Фазі 14.

**Важливе уточнення:** після завершення POM для всіх трьох екранів (login/swipe/forms) три різні spec-файли **успішно** запустились паралельно на тому самому єдиному емуляторі (3 воркери, 3 сесії). Це означає, що конфлікт із попереднього кроку — це не строге правило "ніколи не можна паралельно", а radше гонка за ресурси, чия ймовірність залежить від того, наскільки одночасно й однаково навантажені тести. Висновок не змінюється: паралелізація на одному фізичному пристрої — ризикована, "пощастило" не є гарантією; для надійної паралелізації потрібні окремі пристрої/емулятори (Фаза 14), а не покладання на те, що цього разу не зконфліктувало.

## Фаза 10 — Архітектура фреймворку

### Урок 12: Базовий конфіг + конфіги-розширення по середовищах

**Проблема:** один монолітний `wdio.conf.ts` не масштабується на кілька середовищ (local Android / CI / BrowserStack для iOS, Фаза 13) без дублювання спільних опцій (specs, framework, reporters, hooks).

**Патерн:** базовий конфіг зі спільним, плюс тонкі конфіги-розширення на середовище:
```
wdio.shared.conf.ts   — спільне: specs, framework, reporters, beforeTest/afterTest hooks
wdio.android.conf.ts  — розширює shared: port, capabilities для локального Android, maxInstances
wdio.browserstack.conf.ts (Фаза 13) — розширює shared: hostname, bstack:options, BrowserStack креденшели
```
Розширення через спред об'єкта (`...sharedConfig`) і перевизначення того, що відрізняється. Запуск — явним вказанням файлу: `wdio run ./wdio.android.conf.ts`.

**Виправлення `maxInstances` під реальне обмеження:** shared-конфіг тримає `maxInstances: 10` як загальний дефолт (доречний для BrowserStack, де паралелити можна на кілька хмарних пристроїв), але Android-конфіг **перевизначає на `maxInstances: 1`** — з одним локальним фізичним/віртуальним пристроєм паралельні воркери конфліктують за UiAutomator2 instrumentation-процес (крах, побачений у Фазі 9). Перевизначення в конкретному конфізі середовища, а не в shared, — приклад того, що обмеження паралелізму специфічне для середовища виконання, не універсальне правило проєкту.

## Фаза 11 — Просунуті можливості

### Урок 13: Перемикання контексту (Native ↔ WebView), дозволи, мережеві умови

**Перемикання контексту:**
```ts
const contexts = await driver.getContexts()   // ['NATIVE_APP', 'WEBVIEW_com.wdiodemoapp']
await driver.switchContext(webviewContext)     // тепер $() шукає через CSS/XPath, як у Playwright
await driver.switchContext('NATIVE_APP')
```
**Правило:** ніколи не хардкодити повну назву WEBVIEW-контексту (`WEBVIEW_com.wdiodemoapp`) — вона залежить від package name і може відрізнятись; шукати за префіксом `startsWith('WEBVIEW')` через `getContexts()`.

**Повторна помилка вгадування локатора (навіть у мене):** `$("~Web")` для кнопки навігації — здогадка за видимим текстом, а реальний `content-desc="Webview"` (побачений раніше в Inspector, Урок 6). Підтверджує: перевірка в Inspector потрібна щоразу, досвід не звільняє від цього кроку.

**WebView-контекст не з'являється миттєво після кліку** — та сама тема Фази 8: потрібен `waitUntil()`, що чекає появи `WEBVIEW*` контексту в `getContexts()`, перш ніж перемикатись, інакше `switchContext` наштовхується на власну явну помилку "No WEBVIEW context found" (перевірка спрацювала як задумано, а не крашнулась незрозуміло).

**Інфраструктурне обмеження середовища, знайдене на практиці:** навіть після появи WEBVIEW-контексту, `switchContext` впав з `No Chromedriver found that can automate Chrome '109.0.5414'`. Chromedriver-версії строго прив'язані до конкретної версії Chrome/System WebView на пристрої — застарілий WebView на емуляторі не має відповідного локального chromedriver. Це не помилка коду перемикання контексту (архітектурно правильний), а окрема інфраструктурна проблема середовища (потрібне оновлення System WebView або `chromedriver_autodownload` capability) — задокументовано як відома межа поточного оточення, не вирішувалось негайно, бо немає практичної потреби в реальному WebView-контенті цього demo-застосунку.

**Дозволи (permissions) — мобільна команда, а не клік по системному діалогу:**
```ts
await driver.execute('mobile: changePermissions', {
  permissions: ['android.permission.CAMERA'], action: 'grant', appPackage: APP_PACKAGE,
})
```
Системні діалоги дозволів мають непередбачуваний текст/verstka залежно від locale/версії Android — автоматизувати їх напряму як звичайні UI-елементи крихко.

**Мережеві умови:**
```ts
await driver.execute('mobile: setConnectivity', { wifi: false, data: false })
```

## Фаза 10 (продовження) — Реструктуризація під `src/`

**Причина:** відокремити код самого фреймворку (page objects, конфіги, константи, майбутні утиліти) від тестів — типова структура для production-репозиторіїв, де `src/` = перевикористовуваний код бібліотеки/фреймворку, а `test/` = те, що цей код використовує.

**Нова структура:**
```
src/
  pageobjects/{BasePage,MainScreen,LoginScreen,SwipeScreen,FormsScreen,WebScreen}.ts
  config/{wdio.shared.conf,wdio.android.conf}.ts
  constants.ts
test/
  specs/{login,swipe,forms,web}.e2e.ts
```

**Що довелось виправити після переносу (`git mv`, з історією):**
- Імпорти в spec-файлах: `../pageobjects/...` → `../../src/pageobjects/...` (глибина шляху змінилась).
- `wdio.shared.conf.ts`: імпорт констант `./test/constants.js` → `../constants.js`; `specs`/`tsConfigPath` стали відносними до нового розташування конфігу (`../../test/specs/**/*.ts`, `../../tsconfig.json`) — WebdriverIO резолвить ці шляхи відносно файлу конфігу, що реально виконується, тому переміщення конфігу в підтеку вимагає підняти всі відносні шляхи на два рівні.
- `tsconfig.json`: `include` посилався на вже неіснуючий корінний `wdio.conf.ts` — замінено на `["test", "src"]`.
- `package.json`: скрипт `wdio` тепер вказує на `./src/config/wdio.android.conf.ts`.

**Урок:** переміщення конфігураційного файлу в підтеку — це не просто "перенести файл", а перевірити **кожен відносний шлях усередині нього** окремо, оскільки різні опції (`tsConfigPath`, `specs`, локальні імпорти) можуть мати різні бази відліку (відносно файлу конфігу vs відносно кореня запуску CLI).

## Глосарій (доповнюється)

| Термін | Означення |
|---|---|
| **Appium Server** | Проміжний сервер, що приймає WebDriver-команди і транслює їх у команди платформного драйвера |
| **UiAutomator2** | Офіційний automation-фреймворк Google для Android |
| **XCUITest** | Офіційний automation-фреймворк Apple для iOS |
| **Native app** | Застосунок, що рендериться нативно ОС (Kotlin/Java, Swift/Obj-C), без DOM |
| **Hybrid app** | Нативна оболонка з вбудованим WebView |
| **Context (Appium)** | Режим взаємодії: `NATIVE_APP` (нативне UI-дерево) або `WEBVIEW_*` (DOM усередині WebView) |
| **AVD (Android Virtual Device)** | Конфігурація віртуального Android-пристрою: device profile + system image (API level) + ABI |
| **adb (Android Debug Bridge)** | CLI для взаємодії з Android-пристроєм/емулятором (встановлення APK, логи, shell) |
| **APK** | Файл Android-застосунку для встановлення й тестування |
| **API level** | Числовий номер версії Android (напр. 33 = Android 13), визначає доступні системні можливості |
| **Appium Server** (уточнення) | Окремий Node.js процес, HTTP API за WebDriver протоколом, порт типово 4723 |
| **Driver (Appium)** | Окремий npm-пакет, що реалізує трансляцію команд для конкретної платформи (UiAutomator2, XCUITest) |
| **Session / sessionId** | Ідентифікатор, що прив'язує послідовність команд тесту до конкретного інстансу драйвера й пристрою |
| **Capabilities** | Набір параметрів у `POST /session`, що описує бажану платформу/пристрій/застосунок для нової сесії |
| **Chromium driver (Appium)** | Драйвер, що керує WebView-контекстом через CDP — активується автоматично поруч з uiautomator2 |
| **automationName** | Capability, що вказує Appium, який саме драйвер використати для сесії (напр. `UiAutomator2`, `XCUITest`, `chromium`) |
| **appium:app** | Capability зі шляхом до конкретного встановлюваного файлу застосунку (`.apk`/`.ipa`), не до теки |
| **Phantom dependency** | Пакет, від якого код фактично залежить, але який не вказаний явно в `package.json` — присутній лише транзитивно через іншу залежність |
| **driver / browser (WebdriverIO глобали)** | Глобальні об'єкти сесії; `driver` — природніша назва в мобільному контексті (Appium), `browser` — історична назва з веб-світу, обидві вказують на той самий об'єкт сесії |
| **Accessibility ID (`~`)** | Локатор за `content-desc`, найпріоритетніша й найстабільніша стратегія в Appium |
| **Appium Inspector** | Десктопний GUI для інспекції елементів через живу сесію — аналог Playwright Inspector/Codegen |
| **Lazy element (`$()`)** | Проксі-об'єкт WebdriverIO: фактичний `findElement` відкладений до реальної дії/перевірки, з вбудованим retry |
| **mobile: swipeGesture** | Мобільно-специфічна Appium-команда, делегована нативному UiAutomator2 API жестів (надійніше за ручний W3C Actions) |
| **getLocation() / getSize()** | Методи елемента WebdriverIO для отримання `{x,y}`/`{width,height}` — основа для жестів відносно реального елемента |
| **UiSelector().textContains()** | Android UiAutomator2 DSL-локатор за частковим текстовим вмістом, коли немає власного accessibility id |
| **waitUntil()** | Низькорівневий WebdriverIO-метод очікування довільної булевої умови (не плутати з `expect().toX()`, який кидає виключення, а не повертає false) |
| **getValue() vs getText()** | `getValue()` — спеціалізований метод для input-полів; `getText()` іноді повертає порожній рядок саме на `EditText` |
| **beforeTest / afterTest (WebdriverIO)** | Framework-агностичні глобальні хуки в `wdio.conf.ts`, найближчий аналог Mocha beforeEach/afterEach, але без залежності від конкретного test framework |
| **Page Object singleton** | `export default new ScreenClass()` — сторінка застосунку експортується як готовий екземпляр, не клас |
| **Context (Native/WebView)** | `NATIVE_APP` vs `WEBVIEW_*` — режим інтерпретації локаторів; перемикається через `driver.switchContext()`, список доступних — `driver.getContexts()` |
| **Chromedriver version binding** | Chromedriver суворо прив'язаний до конкретної версії Chrome/System WebView на пристрої — розбіжність версій блокує switchContext на WebView |
