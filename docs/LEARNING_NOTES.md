# Конспект курсу з мобільної автоматизації

Особистий конспект, який веду під час менторства з мобільної автоматизації (Appium 2 + WebdriverIO + TypeScript). Оновлюється після кожного уроку.

---

## Фаза 1 — Основи мобільної автоматизації

### Урок 1: Навіщо існує Appium

**Типи мобільних застосунків:**
- **Native** — Kotlin/Java (Android) або Swift/Obj-C (iOS), рендериться самою ОС, DOM відсутній.
- **Hybrid** — нативна оболонка + WebView усередині (наприклад, checkout-екран).
- **Mobile web** — просто Chrome/Safari на телефоні.

**Чому Playwright-підхід не працює на мобільному:**
Playwright напряму контролює рушії браузера (Chromium/WebKit/Firefox), бо це open-source проєкти з протоколами для зовнішнього керування (CDP). UiAutomator2 (Android) і XCUITest (iOS) — закриті automation-фреймворки, вбудовані в ОС Google/Apple. Немає доступу "всередину" — тільки через офіційні automation API, які надає сама ОС.

**Роль Appium:** сервер, що приймає WebDriver-команди від тесту і **транслює** їх у виклики, специфічні для платформного драйвера (не просте проксіювання — саме трансляція протоколу в інший API).

```
WebdriverIO тест → Appium Server → UiAutomator2 / XCUITest драйвер → Android/iOS automation API → застосунок
```

**Чому мобільна флакі-поведінка гірша за веб:**
На вебі є детерміновані сигнали готовності (DOM ready, network idle, mutation observer). На мобільному таких стандартизованих сигналів немає — UI-дерево будує нативний рендерер ОС, і Appium дізнається про появу елемента лише постфактум, коли він фізично з'явився в дереві. Додатково: анімації переходів, CPU throttling, фонові процеси ОС на реальному пристрої/емуляторі вносять більше змінних, ніж контрольоване середовище headless-браузера. → Тема глибше розкриється у Фазі 8 (синхронізація).

**Навіщо перемикатись у web context на hybrid-екранах:**
Нативні локатори (UiAutomator2) бачать WebView лише як один непрозорий елемент — структури всередині не видно взагалі. Перемикання в web context — це не зручність, а технічна необхідність: без нього немає фізичного доступу до елементів усередині WebView. Після перемикання працюють звичні CSS/XPath-селектори.

**Ключовий висновок:** Appium — не сам automation-рушій, а шар трансляції. Флакі-поведінка й баги найчастіше йдуть від нижчого драйвера (UiAutomator2/XCUITest) або самої ОС, а не від Appium напряму.

## Фаза 2 — Android екосистема

### Урок 2: Складові Android-екосистеми та вибір емулятора

**Ключові частини:**
- **Android SDK** — інструменти для розробки/запуску Android (emulator, adb, платформні образи).
- **Android Studio** — тут використовується як GUI-обгортка над SDK Manager і Device (AVD) Manager.
- **AVD (Android Virtual Device)** — віртуальний пристрій: обираєш device profile, system image (API level), ABI. Аналог `playwright install chromium`, але замість бінарника браузера — цілий віртуальний Android-пристрій конкретної версії ОС.
- **adb (Android Debug Bridge)** — CLI для спілкування з пристроєм/емулятором (встановлення APK, `logcat`, shell). Appium використовує adb під капотом.
- **APK** — файл застосунку для встановлення й тестування (аналог відкриття URL у вебі).
- **Emulator (Android) vs Simulator (iOS)** — НЕ взаємозамінні терміни. Android emulator — справжня віртуалізація (QEMU). iOS simulator — симуляція на рівні API macOS, без повної віртуалізації заліза. Практичний наслідок: нативні бінарні залежності можуть поводитись по-різному.

**Чому для тестового стенду обирають стабільний API level (напр. 33/34), а не найновіший:**
1. **Відповідність реальному розподілу користувачів** — головна причина. Мета тестування — поведінка, з якою реально стикаються користувачі. Тестувати на API, якого ще нема на реальних пристроях, означає перевіряти неіснуючі сценарії і пропускати реальні.
2. **Стабільність драйверів автоматизації** — UiAutomator2/Appium патчаться переважно проти вже випущених API. Preview/найновіші API можуть мати непропатчені баги в accessibility-шарі ОС, які виглядають як флакі тест, а насправді баг платформи.
3. **Відтворюваність у CI** — CI-раннери (напр. `reactivecircus/android-emulator-runner`) мають перевірені, кешовані images для усталених API levels; найновіші/preview можуть бути недоступні або нестабільні в CI.

**Практичний висновок для курсу:** основний AVD для решти курсу — Pixel, API 33 (Android 13), Google APIs image (без Google Play Store, бо Play Services не потрібні для чистої Appium-автоматизації).

**Типові помилки:**
- Плутати emulator (Android) і simulator (iOS) як синоніми.
- Обирати system image навмання, не звертаючи уваги на Google APIs vs Google Play варіант.
- Після зміни конфігурації AVD не перезапускати його повністю (cold restart) — Android Studio попереджає про це банером "Changes will apply on restart".

### Урок 3: adb — базова діагностика

**`adb devices`** — список підключених пристроїв/емуляторів:
```
List of devices attached
emulator-5554   device
```
- `emulator-5554` — serial пристрою (порти емуляторів завжди парні, починаючи з 5554); знадобиться як `udid` capability в Appium-конфігу, коли пристроїв кілька одночасно.
- Статус `device` = готовий. Інші статуси: `offline` (завис), `unauthorized` (не підтверджено USB debugging на реальному пристрої).
- Якщо Appium-конфіг не вкаже явний `udid` при кількох підключених пристроях, буде обрано перший зі списку — часта причина "тест запустився не на тому пристрої" в командних середовищах.

**Формат рядка `logcat`:**
```
07-11 11:17:18.656  1758  1758 I AiAiTextClassifier: повідомлення
   дата+час          PID   TID  рівень   тег              текст
```
Рівні логування: `V < D < I < W < E < F` (Verbose → Fatal), той самий принцип severity, що і в звичайних логерах.

**Ключовий висновок:** повний `logcat` (і навіть відфільтрований по `*:E`) на чистому емуляторі завжди показує велику кількість помилок від системних Google-сервісів (Google Assistant, GMS `constellation`, mock HAL-сервіси на кшталт `power.stats`, `SettingsToPropertiesMapper`) — вони намагаються зв'язатись з хмарою/залізом, якого немає на емуляторі. **Це нормально й не стосується жодного застосунку.**

**Професійне правило:** ніколи не читати весь logcat "як є". Завжди фільтрувати за PID або тегом конкретного тестованого застосунку:
```
adb -s emulator-5554 logcat -d --pid=$(adb -s emulator-5554 shell pidof -s com.example.app)
```
Усе решта — фоновий системний шум, який існує завжди, незалежно від того, що тестується.

## Фаза 3 — Архітектура Appium

### Урок 4: Client-Server архітектура

**Ключова відмінність від Playwright:** у Playwright немає окремого сервера — `chromium.launch()` керує браузером у тому ж/дочірньому процесі прозоро для користувача. В Appium **сервер — окремий, самостійний Node.js процес**, який слухає HTTP-порт (типово `4723`) і має бути запущений *до* старту тестів.

```
Тест (WebdriverIO client) ──HTTP (WebDriver protocol)──▶ Appium Server ──▶ Driver (UiAutomator2/XCUITest) ──▶ Emulator/Device
```

**Appium 2.x архітектура:** сервер — це ядро-диспетчер; драйвери (UiAutomator2, XCUITest) — окремі npm-пакети, встановлюються окремо (`appium driver install uiautomator2`), не вбудовані в сам Appium. Аналог — `playwright install <browser>`, тільки тут йдеться про цілі драйвери автоматизації.

**Сесія (Session):** тест відправляє `POST /session` з `capabilities` (платформа, пристрій, застосунок) → сервер повертає унікальний `sessionId`. Кожна наступна команда (`findElement`, `click`) іде з цим `sessionId` у шляху: `POST /session/{sessionId}/element`. Концептуально схоже на `browser.newContext()` у Playwright, але "контекст" тут — фізичний/віртуальний пристрій.

**Один сервер може обслуговувати кілька паралельних сесій одночасно** — сервер тримає внутрішню мапу `sessionId → інстанс драйвера → конкретний пристрій (udid)` і маршрутизує команди за `sessionId` (аналог session token на бекенді, що асоціює запит із користувачем, а не з конкретним з'єднанням).

**Два підходи до паралелізації:**
| Підхід | Коли використовують |
|---|---|
| 1 сервер, N сесій (мультиплексування через `sessionId`) | Стандарт у CI/хмарних сервісах (BrowserStack, Sauce Labs) — менше накладних витрат |
| N окремих серверів | Потрібна повна ізоляція процесів (різні версії драйверів одночасно) або сервер сам став вузьким місцем через event-loop Node.js при дуже високому навантаженні |

**Типові помилки:**
- Забути запустити Appium Server перед тестами → `ECONNREFUSED`, що новачки плутають з помилкою в тестовому коді.
- Плутати "Appium" як моноліт з "Appium + конкретний драйвер" — проблеми з iOS-елементами зазвичай стосуються XCUITest-драйвера, а не Appium Server як такого.

## Глосарій (доповнюється)

| Термін | Означення |
|---|---|
| **Appium Server** | Проміжний сервер, що приймає WebDriver-команди і транслює їх у команди платформного драйвера |
| **UiAutomator2** | Офіційний automation-фреймворк Google для Android |
| **XCUITest** | Офіційний automation-фреймворк Apple для iOS |
| **Native app** | Застосунок, що рендериться нативно ОС (Kotlin/Java, Swift/Obj-C), без DOM |
| **Hybrid app** | Нативна оболонка з вбудованим WebView |
| **Context (Appium)** | Режим взаємодії: `NATIVE_APP` (нативне UI-дерево) або `WEBVIEW_*` (DOM усередині WebView) |
| **AVD (Android Virtual Device)** | Конфігурація віртуального Android-пристрою: device profile + system image (API level) + ABI |
| **adb (Android Debug Bridge)** | CLI для взаємодії з Android-пристроєм/емулятором (встановлення APK, логи, shell) |
| **APK** | Файл Android-застосунку для встановлення й тестування |
| **API level** | Числовий номер версії Android (напр. 33 = Android 13), визначає доступні системні можливості |
| **Appium Server** (уточнення) | Окремий Node.js процес, HTTP API за WebDriver протоколом, порт типово 4723 |
| **Driver (Appium)** | Окремий npm-пакет, що реалізує трансляцію команд для конкретної платформи (UiAutomator2, XCUITest) |
| **Session / sessionId** | Ідентифікатор, що прив'язує послідовність команд тесту до конкретного інстансу драйвера й пристрою |
| **Capabilities** | Набір параметрів у `POST /session`, що описує бажану платформу/пристрій/застосунок для нової сесії |
